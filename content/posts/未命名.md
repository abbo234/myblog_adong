# 网络基础：面向代理 / VPN / IP 判断的系统版

## 0. 直接结论

网络基础不需要一开始学到企业网络工程师水平。普通用户和中级用户最该掌握的是这几条：

1. **网络通信是分层的**：每一层负责不同事情。故障排查时，要判断问题在哪一层。
    
2. **IP 只解决“从哪到哪”**，不代表账号安全，也不代表隐私完整。
    
3. **DNS 是访问网站前的第一步**，DNS 泄露、污染、劫持会直接影响代理/VPN效果。
    
4. **TCP、UDP、QUIC 会影响速度、延迟、丢包表现**，不是“带宽越大就越快”。
    
5. **HTTPS/TLS 保护内容，不等于隐藏所有访问痕迹**。域名、IP、连接时间、流量大小仍可能暴露。
    
6. **系统代理、TUN、VPN 虚拟网卡不是一回事**，它们接管流量的范围不同。
    
7. **家宽 IP、原生 IP、独享 IP 都不是万能护身符**，真正要看出口 IP 质量、使用历史、ASN、共享程度和目标平台规则。
    

博客原文的价值在于它先讲“信道、协议、协议栈、OSI 分层”等基础概念，这个框架仍然有用；但它没有覆盖足够多的现代现实问题，比如 DoH/DoT/DoQ、HTTP/3、QUIC、TLS 1.3、IPv6、CGNAT、现代代理客户端 DNS 模式、TUN 模式、CDN 与 Anycast 对节点体验的影响。博客中对“信道、带宽、协议、协议栈、分层”的解释仍可作为基础框架使用。([编程随想](https://cirubla.github.io/2021/03/Computer-Networks-Overview.html "计算机网络通讯的【系统性】扫盲——从“基本概念”到“OSI 模型” - 编程随想的博客"))

# 1. 网络通信的最小模型

你访问一个网站，表面上只是打开浏览器输入网址，实际大致经过这条链路：

```text
浏览器 / App
↓
DNS 查询：把域名解析成 IP
↓
本机网络栈：选择走直连、系统代理、TUN、VPN
↓
传输层：TCP / UDP / QUIC
↓
加密层：TLS / HTTPS
↓
目标服务器 / CDN / 代理出口服务器
↓
返回数据
```

如果你用了代理/VPN，中间会多一段：

```text
你的设备
↓
本地代理客户端
↓
代理协议加密传输
↓
远程节点服务器
↓
节点出口 IP 访问目标网站
```

所以排查问题时，不要只问“节点坏了吗”。更准确的问题是：

- DNS 是否解析错？
    
- 浏览器有没有走代理？
    
- 系统代理有没有开启？
    
- TUN 有没有接管流量？
    
- UDP 有没有被支持？
    
- 目标网站是不是屏蔽了出口 IP？
    
- 服务商出口是不是晚高峰拥堵？
    
- 本地网络是不是丢包？

# 2. 分层模型：不要死背 OSI，要用于排障

博客用 OSI 七层模型讲网络，这是经典方式。OSI 七层包括物理层、数据链路层、网络层、传输层、会话层、表示层、应用层；博客也提醒，一般用户重点理解低层到传输层，有助于理解网络与信息安全问题。([编程随想](https://cirubla.github.io/2021/03/Computer-Networks-Overview.html "计算机网络通讯的【系统性】扫盲——从“基本概念”到“OSI 模型” - 编程随想的博客"))

但对代理/VPN实际使用来说，建议用更实用的 **5 层排障模型**：

|实用层级|你要理解什么|常见问题|
|---|---|---|
|物理/接入层|Wi-Fi、网线、4G/5G、宽带|本地没网、信号差、路由器问题|
|IP/路由层|IP 地址、网关、NAT、ASN、路由|某地区节点绕路、IP 被封、路由不稳定|
|传输层|TCP、UDP、QUIC、端口|能 ping 但打不开网页、UDP 不通、游戏/语音异常|
|加密/会话层|TLS、HTTPS、证书、SNI/ECH|证书错误、握手失败、被中间设备干扰|
|应用/代理层|HTTP、DNS、代理协议、规则|订阅失败、规则错、DNS 泄露、App 不走代理|

现实用途：  
以后看到客户端日志，不要只看“error”。要判断它属于哪层：

- `timeout`：可能是路由、端口、节点、目标站问题。
    
- `dns failed`：优先查 DNS。
    
- `tls handshake failed`：查时间、证书、SNI、节点配置。
    
- `connection refused`：可能端口没开、服务端没运行、协议不匹配。
    
- `no route` / `network unreachable`：可能本地网络、TUN、路由表问题。

# 3. IP 基础：IP 是网络身份，不是安全保证

## 3.1 IPv4 与 IPv6

IPv4 是传统地址，例如：

```text
8.8.8.8
```

IPv6 是更长的新地址，例如：

```text
2400:3200::1
```

现实里，IPv4 地址紧缺，所以大量家庭宽带、手机网络、低价服务器都会使用 NAT 或 CGNAT。IPv6 的设计方向是更充分的端到端路由，不应继续依赖 NAT 思路；APNIC 在 2025 年的文章里也明确指出，大规模网络应正确部署 IPv6，而不是继续依赖 NAT 作为长期方案。([APNIC Blog](https://blog.apnic.net/2025/05/13/lets-talk-about-cgnat-and-ipv6-again/ "Let’s talk about CGNAT and IPv6, again | APNIC Blog"))

对你判断代理/VPN的意义：

- IPv4 节点更常见，但 IP 污染、共享、封禁也更常见。
    
- IPv6 节点不一定更好，因为目标网站、客户端、规则、DNS、服务商配置都要支持。
    
- 有些 App 会绕过 IPv4 代理走 IPv6 直连，造成泄露。
    
- 排查时要看：你的真实 IPv4、真实 IPv6、代理出口 IPv4/IPv6 是否一致。
    

## 3.2 公网 IP、私网 IP、NAT、CGNAT

常见私网 IP：

```text
192.168.x.x
10.x.x.x
172.16.x.x - 172.31.x.x
```

这些不是公网 IP，外网不能直接访问你。家庭路由器通常会把多台设备的私网 IP 转换成一个公网出口，这就是 NAT。

CGNAT 是运营商级 NAT。也就是说，不只是你家路由器做 NAT，运营商也在做一层 NAT。结果是：

```text
你的手机 / 电脑
↓
你家路由器 NAT
↓
运营商 CGNAT
↓
公网
```

现实影响：

- 你以为自己有公网 IP，实际可能没有。
    
- BT、远程桌面、NAS、端口映射可能失效。
    
- 游戏、语音、P2P 可能受影响。
    
- 代理服务商说“家宽 IP”，不等于你能拿到完整、干净、可控的公网出口。
    

## 3.3 ASN、ISP、机房、家宽

判断一个代理 IP，不要只看国家地区，要看：

|指标|含义|现实意义|
|---|---|---|
|国家/地区|IP 数据库识别的位置|可能不准，只能参考|
|ASN|IP 属于哪个自治系统|判断是云厂商、运营商、CDN、机房|
|ISP|网络运营商|判断是否像真实家庭宽带|
|类型|数据中心/住宅/移动/教育/企业|影响风控、稳定性、价格|
|历史|是否被滥用、黑名单记录|影响账号登录和网站访问|

关键判断：  
**IP 类型只能说明“出口像什么”，不能保证账号安全。**  
账号平台还会看设备、浏览器指纹、登录位置变化、手机号、支付方式、行为轨迹。服务商把“住宅 IP”包装成“账号绝对安全”，这是典型营销话术。

# 4. DNS 基础：代理/VPN里最容易被忽略的一层

## 4.1 DNS 是什么

DNS 的作用是把域名转换成 IP。比如你输入：

```text
example.com
```

系统需要先查询它对应的 IP，然后才能连接服务器。

常见 DNS 记录：

|记录|作用|
|---|---|
|A|域名对应 IPv4|
|AAAA|域名对应 IPv6|
|CNAME|域名别名|
|MX|邮件服务器|
|TXT|验证、反垃圾邮件、安全策略|
|NS|域名由哪些 DNS 服务器管理|

Cloudflare 的 DNS 资料也把 DNS 解释为互联网的“电话簿”：把人类可读域名转换成机器可读 IP。([Cloudflare](https://www.cloudflare.com/learning/dns/dns-over-tls/ "DNS over TLS vs. DNS over HTTPS | Secure DNS"))

## 4.2 传统 DNS 的问题

传统 DNS 通常是明文的，别人可能看到你查询了什么域名。Cloudflare 的资料明确指出，默认 DNS 查询和响应通常以明文发送，即使网站使用 HTTPS，访问前的 DNS 查询也可能暴露。([Cloudflare](https://www.cloudflare.com/learning/dns/dns-over-tls/ "DNS over TLS vs. DNS over HTTPS | Secure DNS"))

这对代理/VPN非常关键：

```text
网页内容走了代理
但 DNS 查询走了本地运营商
=
目标内容可能看不到，但你查了什么域名可能暴露
```

这就是 DNS 泄露。

## 4.3 DoH、DoT、DoQ

现代 DNS 加密主要有三种：

|类型|全称|大致含义|常见特点|
|---|---|---|---|
|DoH|DNS over HTTPS|DNS 查询封装进 HTTPS|常走 443，像普通 HTTPS 流量|
|DoT|DNS over TLS|DNS 查询通过 TLS 加密|常见端口 853|
|DoQ|DNS over QUIC|DNS 查询通过 QUIC 加密|低延迟、抗丢包设计更现代|

RFC 8484 定义了 DoH：DNS 查询和响应被映射到 HTTP exchange，并通过 HTTPS/TLS 提供完整性和机密性。([RFC 编辑器](https://www.rfc-editor.org/rfc/rfc8484?utm_source=Securitylab.ru "RFC 8484: DNS Queries over HTTPS (DoH) | RFC Editor"))  
RFC 7858 定义了 DoT：用 TLS 为 DNS 提供隐私保护，减少监听和路径篡改风险。([IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc7858?utm_source=chatgpt.com "RFC 7858 - Specification for DNS over Transport Layer ..."))  
RFC 9250 定义了 DoQ：用 QUIC 为 DNS 提供传输机密性，具备类似 DoT 的隐私属性，同时利用 QUIC 降低 TCP 队头阻塞影响。([IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc9250 "RFC 9250 - DNS over Dedicated QUIC Connections"))

现实判断：

- DoH/DoT/DoQ 只能保护 **你到 DNS 解析器之间** 的查询内容。
    
- DNS 解析器本身仍可能知道你查了什么。
    
- 如果代理客户端配置错误，浏览器 DoH 和客户端 DNS 可能打架。
    
- TUN 模式下 DNS 更复杂，必须看客户端是否接管 DNS。
    
- “开了 DoH 就绝对隐私”是错误说法。
    

## 4.4 代理客户端里的 DNS 重点

现代 Clash/sing-box 类客户端通常会涉及：

|项目|意义|
|---|---|
|fake-ip|给域名返回一个虚拟 IP，由客户端内部映射|
|redir-host|返回真实 IP，由规则判断走向|
|nameserver|默认 DNS|
|fallback|备用 DNS|
|proxy DNS|DNS 查询是否走代理|
|direct DNS|DNS 查询是否直连|
|DNS hijack|TUN 下拦截系统 DNS 查询|

普通用户不用一开始深挖所有配置，但必须记住：

**DNS 解析结果会影响你最终连接到哪里。DNS 没处理好，代理开了也可能泄露或打不开。**

最低成本验证：

1. 打开代理。
    
2. 搜索“DNS leak test”。
    
3. 看显示的 DNS 服务器是不是本地运营商。
    
4. 如果仍显示本地运营商，说明可能存在 DNS 泄露。
    
5. 再切换 TUN、系统代理、规则模式对比。

# 5. TCP、UDP、QUIC：为什么有些节点网页快，游戏/语音却烂

## 5.1 TCP

TCP 的特点：

- 面向连接；
    
- 可靠传输；
    
- 丢包会重传；
    
- 顺序交付；
    
- 适合网页、下载、登录、普通应用。
    

缺点：

- 丢包时延迟明显上升；
    
- 建连和重传有成本；
    
- 对实时性要求高的场景可能不如 UDP/QUIC。
    

## 5.2 UDP

UDP 的特点：

- 不保证可靠；
    
- 不保证顺序；
    
- 开销小；
    
- 适合游戏、语音、视频、实时通信、部分代理协议。
    

缺点：

- 容易被网络限制；
    
- 部分机场/节点不支持 UDP；
    
- 公司/校园/公共 Wi-Fi 可能限制 UDP；
    
- UDP 不通时，某些功能会异常，比如游戏、语音、QUIC、部分 DNS。
    

## 5.3 QUIC

QUIC 是现代网络里非常重要的变化。RFC 9000 定义 QUIC 为基于 UDP 的多路复用安全传输协议，提供低延迟连接建立、流控和网络路径迁移等能力。([IETF Datatracker](https://datatracker.ietf.org/doc/rfc9000/ "RFC 9000 - QUIC: A UDP-Based Multiplexed and Secure Transport"))

HTTP/3 就是建立在 QUIC 上。RFC 9114 明确：HTTP/3 是把 HTTP 语义映射到 QUIC 传输协议上，并依赖 QUIC 提供数据机密性、完整性、对端认证、可靠的按流交付等能力。([IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc9114 "RFC 9114 - HTTP/3"))

现实影响：

- 现在很多网站、App、CDN 都可能使用 QUIC/HTTP3。
    
- 如果你的代理/VPN不支持 UDP，HTTP/3 可能不可用，浏览器会回退到 HTTP/2/TCP。
    
- 有些网络会限制 UDP/443，导致 QUIC 表现不稳定。
    
- 某些代理协议如 Hysteria2、TUIC 等依赖 UDP/QUIC 思路，网络环境不支持 UDP 时体验会变差。
    

判断方式：

|现象|可能原因|
|---|---|
|网页能打开，视频卡|UDP、CDN、带宽、晚高峰拥堵|
|浏览器可以，游戏不行|UDP 没走代理或节点不支持 UDP|
|某些 App 慢，网页正常|App 使用 QUIC/UDP，被限制或未代理|
|切换节点后延迟差异大|路由、运营商、出口拥堵不同|

# 6. HTTP、HTTPS、TLS：内容加密不等于全链路隐身

## 6.1 HTTP 是什么

HTTP 是 Web 的基础协议。MDN 对 HTTP 的定义是：它是获取 HTML 等资源的协议，是 Web 数据交换的基础，并且是客户端-服务器协议，请求通常由浏览器发起。([MDN Web Docs](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/Overview "Overview of HTTP - HTTP | MDN"))

通俗说：

```text
HTTP = 浏览器和网站服务器之间说话的规则
```

## 6.2 HTTPS 与 TLS

HTTPS 本质上是：

```text
HTTP + TLS 加密
```

TLS 1.3 是现代 HTTPS 的核心协议之一。RFC 8446 说明，TLS 1.3 用于让客户端/服务器应用在互联网上通信，并防止窃听、篡改和消息伪造。([IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc8446 "RFC 8446 - The Transport Layer Security (TLS) Protocol Version 1.3"))

但要注意：

HTTPS 保护的是你和网站之间的通信内容，不等于隐藏所有元数据。

可能仍然暴露的信息：

- 你连接了哪个 IP；
    
- 连接时间；
    
- 流量大小；
    
- 连接频率；
    
- 部分情况下的域名信息；
    
- DNS 查询；
    
- 代理出口 IP；
    
- 浏览器指纹；
    
- 登录账号行为。
    

## 6.3 SNI 与 ECH

传统 TLS 握手中，SNI 会暴露你要访问的域名。ECH 的目标是加密 ClientHello 中的敏感部分，从而减少中间网络看到具体网站的能力。Mozilla 对 ECH 的解释是：它会加密初始“hello”消息，使中间方更难识别你访问的网站。([Mozilla 支持](https://support.mozilla.org/en-US/kb/understand-encrypted-client-hello?utm_source=chatgpt.com "Understand Encrypted Client Hello (ECH) | Firefox Help"))

但现实判断要保守：

- ECH 不是所有网站都支持。
    
- 客户端、浏览器、DNS、CDN 都要配合。
    
- ECH 不能替代代理/VPN。
    
- ECH 不解决账号风控、浏览器指纹、恶意网站、终端中毒问题。
    

所以不要把 ECH 理解成“新一代万能隐私工具”。

# 7. 带宽、延迟、丢包、抖动：判断节点质量的四个硬指标

博客里对“带宽”的解释仍然有效：带宽指信道单位时间内最大能传输多少比特信息。([编程随想](https://cirubla.github.io/2021/03/Computer-Networks-Overview.html "计算机网络通讯的【系统性】扫盲——从“基本概念”到“OSI 模型” - 编程随想的博客"))

但现实里，用户常犯一个错误：  
**只看带宽，不看延迟、丢包、抖动。**

|指标|含义|对体验的影响|
|---|---|---|
|带宽|最大传输能力|影响下载、视频清晰度|
|延迟|请求往返时间|影响网页响应、游戏、远程操作|
|丢包|数据包丢失比例|影响卡顿、断流、重传|
|抖动|延迟波动|影响语音、视频、游戏稳定性|

现实判断：

- 看视频更吃带宽和丢包。
    
- 打游戏更吃延迟和抖动。
    
- 网页打开速度受 DNS、TCP/TLS握手、CDN、路由影响。
    
- 节点测速高，不代表实际访问目标站快。
    
- 晚高峰慢，通常是服务商线路超售或出口拥堵。
    

最低成本测试：

```text
1. 白天测速一次
2. 晚上 8-11 点测速一次
3. 测 YouTube / 网页 / 下载 / 延迟
4. 对比不同节点
5. 连续观察 3 天
```

不要只信服务商后台测速。那通常只能说明你到节点之间的速度，不等于节点到目标网站也快。

# 8. 路由、CDN、Anycast：为什么“日本节点”不一定快

## 8.1 路由

你访问一个网站，不是直线连接。数据会经过多个运营商、交换节点、国际出口、CDN 节点。

所以：

```text
物理距离近 ≠ 网络路径短
```

一个日本节点可能绕美国，一个香港节点可能晚高峰爆炸，一个美国节点反而因为线路好更稳定。

## 8.2 CDN

很多网站不是从源站直接给你内容，而是通过 CDN 分发。CDN 会根据你的出口 IP、DNS、地理位置、运营商，把你分配到不同边缘节点。

这会导致：

- 同一个网站，不同代理节点看到的服务器不同；
    
- 同一个节点，不同 DNS 解析结果也可能不同；
    
- 流媒体解锁取决于出口 IP、DNS、CDN、账号地区等多因素；
    
- “这个节点能解锁 Netflix”不代表永久能解锁。
    

## 8.3 Anycast

Anycast 是多个服务器共享同一个 IP，由路由系统把你导向“看起来最近/最合适”的节点。博客原文也提到单播、组播、广播、选播/Anycast 等通信方式。([编程随想](https://cirubla.github.io/2021/03/Computer-Networks-Overview.html "计算机网络通讯的【系统性】扫盲——从“基本概念”到“OSI 模型” - 编程随想的博客"))

现实影响：

- 你 ping 某个公共 DNS，不一定是在 ping 同一台机器。
    
- 不同地区访问同一个 IP，可能到达不同机房。
    
- 节点测试结果会受本地运营商路由影响。

# 9. 系统代理、TUN、VPN：普通用户最容易混淆的地方

## 9.1 系统代理

系统代理一般是告诉系统：

```text
支持系统代理的软件，请把流量交给本地代理端口
```

优点：

- 简单；
    
- 浏览器通常有效；
    
- 对小白友好。
    

缺点：

- 不支持系统代理的软件可能直连；
    
- 部分 App 会忽略系统代理；
    
- UDP 通常处理不好；
    
- DNS 可能仍然走本地。
    

## 9.2 TUN 模式

TUN 模式相当于创建一个虚拟网卡，把更多流量接管到代理客户端里。

优点：

- 接管范围更广；
    
- 对不支持系统代理的软件更有效；
    
- 可处理更多全局流量场景。
    

缺点：

- 更容易影响系统网络；
    
- DNS、路由、局域网、游戏、银行软件可能出问题；
    
- 配置错误会导致全局断网；
    
- Windows 下可能受防火墙、驱动、权限影响。
    

## 9.3 VPN 模式

VPN 通常也是通过虚拟网卡接管系统流量。区别是：传统 VPN 更偏系统级隧道，而 Clash/sing-box 这类客户端通常是“代理核心 + 规则路由 + TUN”的组合。

普通用户判断标准：

|使用场景|建议|
|---|---|
|只浏览网页|系统代理通常够用|
|某些 App 不走代理|考虑 TUN|
|游戏/语音/全局接管|需要 TUN/VPN，但要看 UDP|
|银行/政务/本地软件|不建议全局乱代理|
|不懂配置|先不要乱开复杂 DNS/TUN 组合|

# 10. 代理/VPN里的“入口、出口、规则”

现代代理客户端可以理解成三件事：

```text
入口 inbound：本机软件把流量交给客户端的地方
规则 route/rules：客户端判断哪些走代理，哪些直连
出口 outbound：客户端把流量发到哪里
```

例如：

```text
浏览器
↓
127.0.0.1:7890 本地代理入口
↓
规则判断：google.com 走代理，baidu.com 直连
↓
远程节点
↓
节点出口 IP
↓
目标网站
```

这解释了几个常见现象：

|现象|原因|
|---|---|
|开了代理但某软件没效果|软件没有走系统代理，也没被 TUN 接管|
|国内网站变慢|规则把国内网站也代理了|
|国外网站打不开|规则没命中、DNS 错、节点问题|
|测 IP 是代理，但 DNS 是本地|DNS 没被代理接管|
|同一节点不同网站表现不同|目标站/CDN/路由/IP风控不同|

# 11. IP 质量判断：不要被营销词带节奏

服务商常见话术：

|宣传词|实际需要追问|
|---|---|
|原生 IP|哪个 ASN？数据库怎么识别？是否长期稳定？|
|家宽 IP|是否真实 ISP 住宅段？是否共享？是否被多人滥用？|
|独享 IP|独享出口还是独享账号？是否仍然是机房 IP？|
|高速节点|晚高峰怎样？丢包怎样？到目标站怎样？|
|流媒体解锁|解锁哪个平台？哪个区？稳定多久？|
|不限流量|是否限速？是否倍率？是否公平使用政策？|
|永久稳定|基本不可信，网络环境和平台规则都会变|

现实判断：

**IP 质量 = IP 类型 + ASN + 使用历史 + 共享程度 + 目标平台态度 + 用户行为。**

不是：

```text
家宽 = 安全
原生 = 安全
独享 = 安全
贵 = 安全
```

这些都是过度简化。

# 12. 常见误区

## 误区 1：能打开网站 = 安全

错误。  
这只能说明连接成功，不能说明隐私安全、账号安全、DNS 不泄露、服务商不记录日志。

## 误区 2：HTTPS 后就没人知道我访问什么

不完整。  
HTTPS 保护内容，但 DNS、IP、SNI/ECH支持情况、连接元数据、浏览器指纹仍可能暴露。

## 误区 3：TUN 开了就是全局安全

错误。  
TUN 只是更强的流量接管方式。配置错了，可能 DNS 泄露、局域网异常、部分流量直连、全局断网。

## 误区 4：节点测速快就是节点好

错误。  
测速只说明某个测试点到节点的表现。真正要看你访问的目标网站、晚高峰、丢包、路由、出口 IP。

## 误区 5：住宅 IP 一定适合账号登录

错误。  
住宅 IP 只是降低某些平台对“机房 IP”的敏感度，不代表账号行为可信。频繁切换地区、设备异常、行为异常照样可能触发风控。

# 13. 普通用户最低成本验证清单

你以后判断一个代理/VPN服务，先做这些，不要直接年付。

## 13.1 基础连通性

```text
1. 本地网络直连是否正常
2. 客户端是否启动
3. 订阅是否更新成功
4. 节点是否能连接
5. 浏览器是否能打开常见网站
```

## 13.2 IP 检查

```text
1. 打开代理前查一次 IP
2. 打开代理后查一次 IP
3. 看国家/地区/ASN/ISP
4. 看 IPv4 和 IPv6 是否都变了
5. 看是否出现本地真实 IP
```

## 13.3 DNS 检查

```text
1. 做 DNS leak test
2. 看 DNS 是否显示本地运营商
3. 切换系统代理/TUN 对比
4. 检查浏览器是否单独启用了 DoH
5. 检查客户端 DNS 配置
```

## 13.4 速度与稳定性

```text
1. 白天测一次
2. 晚高峰测一次
3. 连续测三天
4. 测网页、视频、下载、延迟
5. 不只看客户端测速数字
```

## 13.5 风险控制

```text
1. 不买长期套餐
2. 不把主力账号直接放到新节点上测试
3. 不使用免费陌生节点登录重要账号
4. 不把订阅链接发给别人
5. 不在不可信客户端导入订阅
6. 不迷信“永久稳定”
```

# 14. 这一部分最终学习顺序

按这个顺序学，最省时间：

```text
1. 网络分层：知道问题发生在哪一层
2. IP/NAT/ASN：知道出口 IP 是什么性质
3. DNS：知道域名解析和泄露问题
4. TCP/UDP/QUIC：知道速度、延迟、丢包为什么不同
5. HTTP/HTTPS/TLS：知道内容加密和元数据暴露边界
6. 路由/CDN/Anycast：知道为什么节点地区不等于实际体验
7. 系统代理/TUN/VPN：知道客户端到底接管了哪些流量
8. IP质量与账号风险：知道服务商宣传不能直接信
```

# 1. 必须补充的核心点

## 1. MTU / MSS：很多“能连但卡、能打开但加载不全”的隐藏原因

**MTU** 是单个网络包最大能多大。代理、VPN、TUN、WireGuard、QUIC、IPv6 都可能额外加封装头，导致包变大。如果路径中某一段承载不了，就可能出现：

- 网页能打开，但图片/视频加载异常；
    
- 某些网站一直转圈；
    
- VPN/TUN 开启后部分 App 不通；
    
- ping 正常，但 HTTPS 卡住；
    
- 移动网络和 Wi-Fi 表现不一致。
    

Windows 的 `netsh interface` 可以查看和配置网络接口参数，微软文档也把它列为管理网络接口、排查连接问题的命令工具。([Microsoft Learn](https://learn.microsoft.com/ja-jp/windows-server/administration/windows-commands/netsh-interface?utm_source=chatgpt.com "netsh interface"))

**现实建议**：  
普通用户不要随便改 MTU。只有在“开代理/TUN/VPN 后部分网站卡死、换网络后恢复、换节点不稳定”时，才把 MTU 当作排查项。

---

## 2. IPv6 泄露：现在比旧教程时代更重要

旧博客时代 IPv6 普及度没现在高。现在很多宽带、手机网络、系统、浏览器都可能启用 IPv6。问题是：你的 IPv4 走代理了，但 IPv6 可能直连。

典型现象：

```text
查 IPv4：显示代理节点
查 IPv6：显示真实运营商
```

这就是泄露风险。

**现实建议**：

- 开代理后同时检查 IPv4 和 IPv6；
    
- 如果客户端不处理 IPv6，要么正确配置代理客户端，要么临时关闭系统 IPv6；
    
- 不要只用一个“查 IP 网站”判断是否安全；
    
- TUN 模式下尤其要确认 IPv6 是否被接管。
    

---

## 3. 浏览器 DoH 与代理客户端 DNS 可能冲突

现在 Chrome、Edge、Firefox 都支持“安全 DNS / DoH”。Cloudflare 的文档中也列出 Chrome、Edge 可以在设置中启用 Use secure DNS，并选择 DoH 提供商。([Cloudflare Docs](https://developers.cloudflare.com/1.1.1.1/encryption/dns-over-https/encrypted-dns-browsers/?utm_source=chatgpt.com "Configure DoH on your browser"))

问题是：  
你代理客户端里设置了一套 DNS，浏览器自己又开了一套 DoH，最后可能出现：

- 浏览器 DNS 不走代理客户端；
    
- 规则分流失效；
    
- DNS 泄露检测结果混乱；
    
- 某些网站解析到错误地区；
    
- 流媒体解锁异常。
    

Google Public DNS 文档也说明 DoH 通过 HTTPS API 提供 DNS 查询能力，而 DoT 是通过 TLS 加密 TCP 连接进行 DNS 解析。([Google for Developers](https://developers.google.com/speed/public-dns/docs/doh?utm_source=chatgpt.com "DNS-over-HTTPS (DoH) | Public DNS"))

**现实建议**：

如果你用 Clash / sing-box / Clash Verge Rev 这类客户端，优先让客户端统一处理 DNS。浏览器 DoH 不一定要开，尤其是排查阶段应先关闭，避免变量太多。

---

## 4. WebRTC 泄露：不是所有 IP 风险都来自代理本身

WebRTC 用于浏览器实时通信。它的 ICE 机制可能暴露局域网地址、CGNAT 地址或某些网络元数据。IETF 的 mDNS ICE 草案说明，WebRTC 收集 ICE candidates 时可能包含客户端私有 IP 地址，因此引入 mDNS 主机名替代 IP 来减少隐私暴露。([IETF](https://www.ietf.org/archive/id/draft-ietf-mmusic-mdns-ice-candidates-02.html?utm_source=chatgpt.com "Using Multicast DNS to protect privacy when exposing ICE ..."))

这说明一个现实问题：  
**即使你代理开了，浏览器本身也可能通过 WebRTC 暴露额外网络信息。**

但也不要过度恐慌。现代浏览器已经做了很多缓解，不像早期那么容易直接暴露真实公网 IP。真正该做的是验证，而不是玄学焦虑。

**现实建议**：

- 用 WebRTC leak test 检查；
    
- 浏览器别乱装未知插件；
    
- 登录重要账号时，不要频繁切换代理地区；
    
- 高风险隐私场景用专门浏览器配置，不要混用日常浏览器。
    

---

## 5. 证书信任与中间人风险：免费节点尤其要小心

HTTPS/TLS 保护内容，但前提是：

- 证书链正常；
    
- 没有安装恶意根证书；
    
- 浏览器没有忽略证书警告；
    
- 客户端和系统没有被劫持。
    

很多人只关心“节点能不能用”，忽略了：  
如果你用了不可信客户端、来路不明的证书、免费节点、被篡改安装包，风险可能不在网络协议，而在你的设备和信任链。

**现实建议**：

- 不要安装陌生人发来的“专用客户端”；
    
- 不要随便安装根证书；
    
- 浏览器提示证书错误，不要强行继续；
    
- 下载客户端优先 GitHub Release / 官网；
    
- 订阅链接不要导入到不可信客户端。

# 2. 进阶但重要的补充点

## 6. NAT 类型与 UDP：影响游戏、语音、P2P、部分代理协议

NAT 不只是“能不能上网”。不同 NAT 类型会影响 UDP 打洞、游戏联机、语音通话、P2P、部分代理协议。

现实表现：

|现象|可能原因|
|---|---|
|网页正常，游戏联机失败|UDP/NAT 问题|
|语音通话卡顿|UDP 丢包、NAT、QoS|
|Hysteria2/TUIC 不稳定|UDP 被限或丢包高|
|WireGuard 连不上|UDP 被阻断或端口被拦|

QUIC 本身就是基于 UDP 的安全传输协议，IETF RFC 9000 说明 QUIC 提供低延迟连接建立、流控和网络路径迁移。([IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc9000?utm_source=chatgpt.com "RFC 9000 - QUIC: A UDP-Based Multiplexed and Secure ..."))

**现实建议**：  
如果你只浏览网页，UDP 不一定重要；如果你用游戏、语音、视频会议、Hysteria2、TUIC、WireGuard，UDP 就很重要。

---

## 7. MASQUE / CONNECT-UDP：代理技术正在向 HTTP/3/QUIC 演进

这是进阶内容，不适合小白一开始深挖，但有必要知道趋势。

IETF RFC 9298 定义了通过 HTTP 代理 UDP 的机制，也就是 CONNECT-UDP，属于 MASQUE 相关技术方向。它允许 HTTP 客户端通过 HTTP 服务器建立 UDP 隧道。([IETF Datatracker](https://datatracker.ietf.org/doc/rfc9298/?utm_source=chatgpt.com "RFC 9298 - Proxying UDP in HTTP"))

现实意义：

- 未来代理/VPN技术会更多借助 HTTP/3、QUIC、MASQUE；
    
- “代理”和“VPN”的边界会越来越模糊；
    
- 但普通用户不要被新名词带节奏；
    
- 真正要看：稳定性、客户端成熟度、服务商实现质量、是否适合你的网络环境。
    

**结论**：  
MASQUE 是技术趋势，不是你现在买服务时的核心判断标准。

---

## 8. ODoH：DNS 隐私增强，但不是万能

ODoH 是 Oblivious DNS over HTTPS。RFC 9230 说明，它通过代理加密 DNS over HTTPS 消息，使单一服务器不能同时知道客户端 IP 和 DNS 查询内容。([IETF Datatracker](https://datatracker.ietf.org/doc/rfc9230/?utm_source=chatgpt.com "RFC 9230 - Oblivious DNS over HTTPS"))

这比普通 DoH 更进一步，但现实中要冷静：

- ODoH 不是所有系统和客户端默认支持；
    
- 它保护的是 DNS 查询隐私，不保护全部网络流量；
    
- DNS 隐私增强不等于账号安全；
    
- 解析器、代理、目标网站、浏览器指纹仍是不同风险面。
    

**结论**：  
ODoH 属于隐私增强技术，不是普通代理用户的第一优先级。

---

## 9. Captive Portal：公共 Wi-Fi 登录页会干扰代理/VPN

公共 Wi-Fi 常见“认证页”：

```text
酒店 Wi-Fi
商场 Wi-Fi
工厂宿舍 Wi-Fi
校园 Wi-Fi
机场 Wi-Fi
```

这类网络可能要求你先打开一个登录页。如果你一上来就开代理/TUN/VPN，可能导致：

- 登录页打不开；
    
- DNS 被劫持到认证页面；
    
- HTTPS 报错；
    
- VPN 连不上；
    
- 以为节点坏了，其实是 Wi-Fi 还没认证。
    

**现实建议**：

连接公共 Wi-Fi 后，先不开代理，打开任意 HTTP 网站触发认证，确认能正常上网后再开代理。

---

## 10. 日志阅读能力：比盲目重装客户端更重要

代理客户端日志里常见关键词要能看懂：

|日志关键词|大致含义|优先检查|
|---|---|---|
|`timeout`|超时|节点、网络、目标站|
|`connection refused`|对方拒绝连接|端口、服务端、协议|
|`no such host`|DNS 解析失败|DNS 设置|
|`tls handshake failed`|TLS 握手失败|时间、证书、SNI、配置|
|`network unreachable`|网络不可达|本地网络、路由、TUN|
|`i/o timeout`|读写超时|丢包、节点拥堵、目标站|
|`403/429`|被目标站拒绝或限流|IP 质量、访问频率、风控|

**现实建议**：  
以后排障时先看日志，不要第一反应就是“卸载重装”。重装解决的是少数客户端损坏问题，不解决 DNS、路由、节点、出口 IP、目标站限制。

# 还应补充的 9 个知识点

## 1. Happy Eyeballs：为什么 IPv6/IPv4 会自动切换

现代浏览器和系统通常不会死等 IPv6 或 IPv4，而是会用类似 **Happy Eyeballs** 的机制，尝试减少用户看到的连接延迟。RFC 8305 说明，它的目标就是减少连接失败或等待造成的可见延迟。([IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc8305?utm_source=chatgpt.com "RFC 8305 - Happy Eyeballs Version 2: Better Connectivity ..."))

现实影响：

- 你以为自己只走 IPv4，实际系统可能优先尝试 IPv6；
    
- 某些网站有时快、有时慢，可能和 IPv4/IPv6选择有关；
    
- 代理只处理 IPv4、不处理 IPv6 时，可能出现泄露或连接异常；
    
- 查 IP 时必须同时查 IPv4 和 IPv6。
    

---

## 2. EDNS Client Subnet：DNS 也可能暴露大致网络位置

ECS，也就是 **EDNS Client Subnet**，会在 DNS 查询里携带发起查询网络的部分前缀信息，用于让 CDN 做更准确的调度；RFC 7871 同时也说明它存在已知的运营和隐私缺陷。([IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc7871?utm_source=chatgpt.com "RFC 7871 - Client Subnet in DNS Queries"))

现实影响：

- 公共 DNS 不一定完全隐藏你的位置；
    
- CDN 可能根据 DNS 查询来源给你分配不同节点；
    
- 流媒体解锁、网站地区识别不只看出口 IP，也可能受 DNS 路径影响；
    
- “换 DNS 就能稳定解锁”是过度简化。
    

---

## 3. DNS 缓存：改了设置不一定立刻生效

系统、浏览器、代理客户端、路由器、公共 DNS、CDN 都可能缓存 DNS 结果。

现实影响：

- 你切换节点后，网站仍然解析到旧 IP；
    
- 你改了 DNS，但浏览器还在用旧缓存；
    
- 某些网站突然打不开，可能不是节点坏，而是缓存了错误解析；
    
- 排查 DNS 时，最好重启浏览器、刷新客户端 DNS、必要时清理系统 DNS 缓存。
    

这一点不用深学原理，但要知道它会制造“我明明改了，为什么没变化”的错觉。

---

## 4. 防火墙和杀毒软件：很多 TUN/VPN 问题不是代理配置问题

Windows 防火墙会按网络类型、应用、入站/出站规则过滤流量。微软文档说明，Windows 安全中心可以查看和管理防火墙状态，并按域网络、专用网络、公用网络分别设置。([微软支持](https://support.microsoft.com/en-us/windows/firewall-and-network-protection-in-the-windows-security-app-ec0844f7-aebd-0583-67fe-601ecf5d774f?utm_source=chatgpt.com "Firewall and Network Protection in the Windows Security App"))

现实影响：

- Clash Verge Rev / sing-box / v2rayN 可能被防火墙拦截；
    
- TUN 虚拟网卡可能被安全软件干扰；
    
- 更新客户端后路径变了，旧的防火墙放行规则可能失效；
    
- 不建议直接关闭防火墙，优先给可信客户端单独放行。
    

---

## 5. 系统时间：时间错了会导致 TLS、登录、证书异常

Windows 时间服务对安全认证很重要，微软文档明确说明 W32Time 对 Kerberos 认证和基于 AD DS 的认证很关键。([Microsoft Learn](https://learn.microsoft.com/en-us/windows-server/networking/windows-time-service/windows-time-service-tech-ref?utm_source=chatgpt.com "Windows Time Service Technical Reference"))

现实影响：

- 系统时间不准，可能出现 HTTPS 证书错误；
    
- 登录网站、同步订阅、连接节点时可能失败；
    
- 某些客户端日志会出现 TLS 握手异常；
    
- 排查证书类错误时，先看系统时间、时区、日期是否正确。
    

这类问题很低级，但真实发生时很浪费时间。

---

## 6. HSTS：浏览器会强制某些网站走 HTTPS

HSTS 会告诉浏览器，以后访问该域名必须使用 HTTPS；MDN 说明，启用 HSTS 后，浏览器会把后续 HTTP 请求自动升级为 HTTPS，并且不允许用户绕过无效证书错误。([MDN Web Docs](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Strict-Transport-Security?utm_source=chatgpt.com "Strict-Transport-Security header - HTTP - MDN Web Docs"))

现实影响：

- 某些公共 Wi-Fi 劫持登录页时，HTTPS 网站会报错或打不开；
    
- 证书错误不能随便点“继续访问”；
    
- 如果代理、DNS、公共 Wi-Fi 认证页互相冲突，会出现看似奇怪的连接失败；
    
- HSTS 是安全机制，不是代理问题。
    

---

## 7. 浏览器状态隔离：IP 不是唯一追踪维度

Firefox 的 Total Cookie Protection 会把不同网站的 Cookie 分开，减少跨站追踪；Firefox 官方页面说明，它会让每个网站的 Cookie 保持隔离，防止第三方追踪跨站跟随。([Firefox](https://www.firefox.com/en-US/features/total-cookie-protection/?utm_source=chatgpt.com "Total Cookie Protection"))

现实影响：

- 平台识别你，不只靠 IP；
    
- Cookie、LocalStorage、浏览器指纹、登录历史、设备环境都可能参与识别；
    
- 换代理不等于换身份；
    
- 用同一个浏览器、同一套 Cookie、同一个账号频繁切节点，反而更容易异常。
    

对普通用户的结论：  
**不要把代理/VPN当作账号风控万能解决方案。**

---

## 8. 浏览器级 IP 保护：以后“浏览器自带代理”会更多

Google Privacy Sandbox 的 IP Protection 介绍说，它会在无痕模式下通过双跳代理系统匿名化符合条件的流量，用来降低第三方上下文中的 IP 暴露。([Privacy Sandbox](https://privacysandbox.google.com/protections/ip-protection?utm_source=chatgpt.com "IP Protection Overview | Privacy Sandbox - Google"))

现实影响：

- 未来浏览器可能自己带部分代理/隐私转发能力；
    
- 这会让“系统代理、浏览器代理、客户端代理”之间更复杂；
    
- 排查时要看浏览器是否启用了额外隐私代理功能；
    
- 这类功能通常不是完整 VPN，不要误解为全局代理。
    

---

## 9. 免费代理风险：不是“不稳定”这么简单

2024 年一项对免费代理的长期研究收集并测试了大量免费代理，发现免费代理普遍不稳定，并且存在内容篡改、漏洞主机、恶意行为等风险。研究中只有一部分代理在测试期间至少活跃过，还发现部分代理会操纵内容。([arXiv](https://arxiv.org/abs/2403.02445?utm_source=chatgpt.com "Free Proxies Unmasked: A Vulnerability and Longitudinal Analysis of Free Proxy Services"))

现实判断：

- 免费代理最大问题不是慢，而是你不知道谁在控制出口；
    
- 不要用免费代理登录主力账号；
    
- 不要通过免费代理访问支付、邮箱、网盘、工作系统；
    
- 免费节点适合低风险连通性测试，不适合长期使用。
