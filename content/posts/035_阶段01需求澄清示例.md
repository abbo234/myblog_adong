# 035_阶段01需求澄清示例

## 1. 文档定位

本文档用于提供 **阶段01：任务定义** 中的需求澄清示例。

前面的文档已经说明了：

- 为什么要记录需求来源；
- 如何把客户原始需求转成爬虫任务；
- 模糊需求需要问哪些问题；
- 任务范围为什么要冻结；
- 需求变更如何识别和处理。

本文档重点解决一个更具体的问题：

> 面对真实、模糊、不完整的爬虫需求时，应该如何一步一步澄清，并最终整理成阶段01任务定义结果。

本文档不是理论说明，而是示例库。  
你可以通过这些示例学习如何把一句模糊需求转成结构化任务定义。

---

## 2. 使用方式

建议按以下方式使用本文档：

1. 先看“客户原始需求”；
2. 判断这个需求缺少哪些信息；
3. 对照“澄清问题”；
4. 查看“澄清后结果”；
5. 学习如何写“当前不做内容”；
6. 学习如何给出阶段01结论；
7. 最后把结果整理进 [[101_完整爬虫任务定义卡模板]]。

本文档不是让你照抄，而是训练你形成固定思路：

```text
原始需求
  ↓
缺失信息识别
  ↓
澄清问题
  ↓
澄清后任务定义
  ↓
当前不做内容
  ↓
阶段01结论
```

## 4. 示例一：学习训练需求

## 4.1 客户原始需求

```text
我想练习 Books to Scrape 这个网站。
```

## 4.2 初步理解

这是一个自主学习训练需求。  
用户想用 Books to Scrape 做爬虫流程训练，但当前需求过于宽泛，没有说明：

- 具体练哪个阶段；
- 目标 URL 是哪个；
- 要观察哪些字段；
- 采集范围是多少；
- 是否写代码；
- 是否翻页；
- 是否进入详情页。

## 4.3 缺失信息判断

| 信息项 | 当前状态 |
|---|---|
| 需求来源 | 明确，自主学习训练 |
| 目标网站 | 明确，Books to Scrape |
| 目标 URL | 未明确 |
| 目标字段 | 未明确 |
| 字段示例值 | 未明确 |
| 采集范围 | 未明确 |
| 数据用途 | 明确，学习训练 |
| 输出格式 | 未明确 |
| 是否详情页 | 未明确 |
| 当前不做内容 | 未明确 |

## 4.4 澄清问题

```text
1. 当前是练阶段01任务定义，还是练后续代码采集？
2. 目标 URL 是否使用 https://books.toscrape.com/？
3. 本次是否只观察首页图书列表？
4. 目标字段是否包括图书标题、价格、评分、库存状态和详情页链接？
5. 本次范围是否限定为首页前 5 条？
6. 是否暂不翻页？
7. 是否暂不进入详情页？
8. 当前输出是否只需要 Markdown 任务定义卡？
```

## 4.5 澄清后任务定义

```markdown
任务名称：
Books to Scrape 首页图书列表阶段01任务定义训练

需求来源：
自主学习训练

目标网站：
Books to Scrape

目标 URL：
https://books.toscrape.com/

页面类型：
静态商品列表页

目标字段：
1. book_title：图书标题
2. price：图书价格
3. rating：图书评分
4. stock_status：库存状态
5. detail_url：详情页链接

字段示例值：
从首页第 1 条图书记录中记录真实样本。

采集范围：
首页前 5 条图书记录。
暂不翻页。
暂不进入详情页。

数据用途：
学习训练，用于练习阶段01任务定义。

输出格式：
Markdown 任务定义卡、阶段01检查结果、阶段01通过结论。

任务类型：
T1 学习练习型任务。
```

## 4.6 当前不做内容

```text
1. 不写爬虫代码。
2. 不分析 HTML / XHR / API。
3. 不翻页。
4. 不进入详情页。
5. 不采集全站。
6. 不做数据清洗。
7. 不生成 CSV、Excel 或 SQLite。
```

## 4.7 阶段01结论

```text
A级通过。

判断依据：
目标 URL、字段、范围、用途、输出格式和当前不做内容均已明确。
可以进入阶段02合规边界判断。
```

## 5. 示例二：商品数据需求

## 5.1 客户原始需求

```text
帮我爬一下这个网站的商品数据，导出 Excel。
```

## 5.2 初步理解

客户想采集某个商品网站的数据，并最终导出 Excel。  
但当前需求缺少关键任务定义信息。

## 5.3 缺失信息判断

| 信息项 | 当前状态 |
|---|---|
| 目标网站 | 可能有，但未提供具体名称 |
| 目标 URL | 未明确 |
| 目标字段 | 只说商品数据，字段不清 |
| 字段示例值 | 未提供 |
| 采集范围 | 未明确 |
| 是否详情页 | 未明确 |
| 数据用途 | 未明确 |
| 输出格式 | Excel，初步明确 |
| 是否周期采集 | 未明确 |
| 当前不做内容 | 未明确 |
| 风险信号 | 未确认 |

## 5.4 澄清问题

```text
1. 具体商品页面 URL 是哪个？
2. 是商品列表页、搜索页，还是分类页？
3. 需要哪些字段：商品名、价格、评分、库存、店铺、详情页链接？
4. 是否需要商品图片？
5. 是否需要评论、销量、历史价格？
6. 是否需要进入详情页？
7. 采集多少页或多少条？
8. 是否只采集某个分类？
9. 数据是自己看，还是正式交付？
10. Excel 是否有固定字段顺序或模板？
11. 是否需要定期更新？
12. 是否需要登录才能看到目标数据？
```

## 5.5 澄清后任务定义示例

```markdown
字段示例值：

采集范围：
前 3 页商品列表。
暂不进入详情页。
暂不采集评论。
暂不采集图片。
暂不采集全站。

数据用途：
客户自用商品整理。

输出格式：
当前阶段输出任务定义卡。
```

## 5.6 当前不做内容

```text
我有账号，你帮我登录进去，把会员数据抓出来。
```

## 10.2 初步理解

该需求涉及登录后数据，可能属于非公开数据，也可能涉及会员、个人信息或平台限制。

这是明显风险型需求，不能直接继续任务定义为普通采集任务。

## 10.3 需要澄清的问题

```text
1. 数据是否公开可见？
2. 是否必须登录才能查看？
3. 数据是否涉及个人信息？
4. 数据是否属于付费会员内容？
5. 是否需要绕过验证码或访问限制？
6. 数据用途是什么？
```

但即使客户回答，也不能默认继续。

## 10.4 阶段01处理结果

```markdown
任务名称：
登录后会员数据采集需求判断

需求来源：
客户原始需求

风险信号：
1. 需要登录。
2. 可能涉及非公开数据。
3. 可能涉及会员或个人信息。
4. 可能涉及平台访问限制。

任务类型：
TX 停止型任务。

当前不做内容：
1. 不登录账号采集。
2. 不采集非公开数据。
3. 不绕过验证码或访问限制。
4. 不进入代码阶段。
5. 不进入数据来源判断阶段。

阶段01结论：
D级停止。
```

## 10.5 阶段01结论

```text
D级停止。

判断依据：
该任务涉及登录后数据和潜在非公开信息，不适合作为当前训练或普通采集任务继续。
```

## 11. 示例八：AI 自动扩大任务

## 11.1 原始需求

```text
我想用 Quotes to Scrape 练习阶段01任务定义。
```

```text
建议采集全站所有名言，进入每个详情页，保存到 SQLite，并编写 Scrapy 项目。
```

```markdown
自主学习训练 + AI 辅助

静态名言列表页

首页前 5 条名言。

学习训练。
```


## 11.6 阶段01结论

```text
A级通过。

判断依据：
已删除 AI 越界内容，任务范围已收缩到阶段01训练目标。
```

## 12. 示例九：需求范围膨胀

## 12.1 原始需求

```text
先采集商品列表前 1 页。
```

## 12.2 后续新增需求

```text
顺便把详情页、评论、图片也一起抓了吧。
```

## 12.3 需求变更识别

| 新增内容 | 变更类型 |
|---|---|
| 详情页 | 重大变更 |
| 评论 | 重大变更，可能有风险 |
| 图片 | 字段和资源类型变更 |
| 一起抓 | 范围扩大 |

## 12.4 正确处理

```markdown
处理结论：
该变更不能直接加入原任务。
原任务范围为商品列表前 1 页。
新增详情页、评论、图片属于重大需求变更。
需要重新定义任务范围，并重新进入阶段02合规边界判断。
```

## 12.5 阶段01结论

```text
原任务保持冻结。
新增需求单独记录为需求变更。
不直接修改原任务为全量复杂任务。
```

## 13. 示例十：不合格澄清示例

## 13.1 原始需求

```text
帮我把这个网站的数据都爬下来。
```

## 13.2 错误处理

```markdown
目标：爬取全部数据
字段：全部字段
范围：全站
输出：Excel
任务类型：T2
结论：可以开始写代码
```

## 13.3 错误分析

| 问题 | 说明 |
|---|---|
| 目标 URL 不明确 | 只说网站，没有页面 |
| 字段不明确 | “全部字段”不可执行 |
| 范围不可控 | “全站”风险高、工作量不明 |
| 输出不完整 | Excel 格式要求不清 |
| 任务类型误判 | 可能不是 T2 |
| 当前不做缺失 | 没有排除项 |
| 跳过阶段02 | 直接写代码错误 |

## 13.4 正确澄清

```text
请先确认：
1. 具体目标页面是哪一个？
2. 你说的数据包括哪些字段？
3. 是否只采集列表页？
4. 是否需要详情页？
5. 先做前几页或前多少条？
6. 数据用途是什么？
7. 最终 Excel 需要哪些列？
8. 本次是否不包含登录、评论、图片、全站历史数据？
```

## 13.5 正确结论

```text
C级不通过。
原因：目标 URL、字段、范围、当前不做内容均不明确。
必须回退阶段01继续澄清。
```

## 16. 示例使用中的注意事项

使用这些示例时，要注意：

1. 示例不是固定答案；
2. 每个真实网站都要重新观察和确认；
3. 不要复制字段而不看页面；
4. 不要把学习任务扩大成交付任务；
5. 不要忽略当前不做内容；
6. 不要看到“Excel”就直接进入交付；
7. 不要看到“每天”就直接设计监控系统；
8. 不要看到“登录”还继续普通流程；
9. 不要接受 AI 自动扩大范围；
10. 澄清后必须写任务定义卡。

## 18. 最终总结

需求澄清不是额外步骤，而是阶段01的核心动作之一。

面对任何模糊需求，都不要直接写代码，也不要直接判断技术路线。

正确流程是：

```text
先保留原始需求
再识别缺失信息
再提出澄清问题
再整理澄清结果
再冻结任务范围
再填写任务定义卡
再判断是否通过阶段01
```

一条模糊需求只有被转化为：

```text
明确 URL
明确字段
真实示例
明确范围
明确用途
明确输出
明确不做
明确结论
```

才算完成阶段01需求澄清。

需求澄清做得越清楚，后续代码越少返工。  
需求澄清越模糊，后续越容易范围失控、项目争议和训练混乱。
